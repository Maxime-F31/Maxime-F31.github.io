Réseau Mesh Ad-Hoc — ONERA Toulouse

Janvier – Février 2026 · Stage de 5 semaines

Recherche GitHub

Développement et test d'applications sur des Raspberry Pi Zero 2 W au sein d'un réseau mesh ad-hoc, coordonné par un nœud maître (Raspberry Pi 3). L'objectif était d'étudier la gestion du trafic et la génération de données dans un réseau dynamique multi-nœuds.

Raspberry PiBATMAN-advOLSRv2 BashIBSS 802.11Debian Bookworm JSONChrony NTP

L'ONERA (Office National d'Études et de Recherches Aérospatiales) est le principal organisme public français de recherche dans les domaines aéronautique, spatial et de défense, créé en 1946. Le centre de Toulouse, situé sur le campus de Rangueil, se spécialise dans le traitement de l'information et les systèmes embarqués.

① Surveiller le réseau mesh en temps réel (état des nœuds, liaisons, trafic)
② Générer du trafic applicatif pour tester les performances (ping, latence, débit)
③ Comparer les protocoles de routage OLSR et BATMAN-adv

Matériel : Raspberry Pi Zero 2 W (nœuds) + Raspberry Pi 3 (nœud maître). Choisis pour leur faible consommation, leur flexibilité et la possibilité de faire tourner un véritable Linux. L'ESP32 a été écarté car il ne supporte pas nativement le mode IBSS 802.11.

OS : Debian Bookworm, retenu pour sa stabilité, sa compatibilité Raspberry Pi éprouvée et sa documentation abondante. Trixie (trop récent) et Ubuntu Server (problèmes Imager) ont été écartés.

Un script Bash complet (setup-mesh-batman.sh) automatise l'intégralité de la configuration :

Bureau (OLSR)

Toutes les puces à portée directe. Prise de valeurs de référence.

✓ Succès — Tous les nœuds communiquent

Sous-sol (OLSR)

Topologie en bus. Les pings ne passent pas à N+1 sauts.

✗ Échec — Problème de routage OLSR

Sous-sol (BATMAN-adv)

Le routage fonctionne mais la latence augmente sous charge.

~ Partiel — Latence élevée

Mix (BATMAN-adv)

Ligne droite avec nœud mobile. Correctifs du JSON.

✓ Tests OK

Réseaux : Configuration IBSS, protocoles mesh, routage multi-sauts, analyse de latence
Systèmes : Administration Linux, services systemd, scripting Bash avancé, NTP
Embarqué : Déploiement Raspberry Pi, contraintes matérielles, modules kernel
Méthodologie : Journalisation JSON, tests reproductibles, documentation technique

Maquette Infrastructure Réseau — BTS SIO

2024 – 2026 · Épreuve obligatoire BTS SIO SISR

Épreuve BTS

Dans le cadre du BTS SIO, la mise en place d'une maquette d'infrastructure réseau complète est obligatoire pour l'obtention du diplôme. Une entreprise fictive est créée afin de simuler des besoins réels auxquels nous devons apporter des solutions techniques concrètes. L'objectif est de démontrer notre capacité à concevoir, déployer et administrer une infrastructure réseau d'entreprise fonctionnelle.

ProxmoxpfSenseActive Directory OpenVPNZabbixGrafana FreePBXAsteriskWindows Server

Hyperviseur : Un serveur Proxmox héberge l'ensemble des machines virtuelles de l'infrastructure (sauf l'Active Directory).

Physique : Un ordinateur Windows dédié pour le serveur Active Directory, connecté via un switch physique au réseau. Des téléphones IP sont également branchés directement sur ce switch.

Réseau (pfSense) : Un pare-feu pfSense virtuel gère trois zones réseau distinctes :
  • WAN — Simule Internet et les connexions VPN des collaborateurs distants
  • LAN — Héberge les clients Windows, l'Active Directory, le serveur FreePBX/Asterisk
  • DMZ — Contient le serveur de supervision Zabbix + Grafana et le serveur OpenVPN

Active Directory

Serveur Windows Server assurant l'annuaire centralisé, la gestion des utilisateurs, des groupes et des stratégies de groupe (GPO) pour l'ensemble du domaine de l'entreprise.

✓ Déployé sur machine physique

OpenVPN + Pritunl

Serveur VPN permettant aux collaborateurs externes de se connecter à distance à l'infrastructure de l'entreprise de manière sécurisée, hébergé en DMZ.

✓ Accès distant fonctionnel

Zabbix + Grafana

Solution de supervision réseau permettant de monitorer en temps réel l'état de tous les équipements et services du réseau. Tableaux de bord visuels via Grafana.

✓ Supervision active en DMZ

FreePBX + Asterisk

Serveur de téléphonie IP permettant aux collaborateurs de s'appeler directement sur le réseau interne de l'entreprise via des téléphones IP physiques.

✓ Appels internes opérationnels

pfSense

Pare-feu et routeur virtuel assurant le cloisonnement réseau (WAN/LAN/DMZ), le NAT, les règles de filtrage et la gestion du trafic entre les zones.

✓ 3 zones configurées

Proxmox VE

Hyperviseur de type 1 hébergeant l'ensemble des machines virtuelles. Permet la gestion centralisée, les snapshots et la scalabilité de l'infrastructure.

✓ Hébergement de toutes les VM

Cette maquette répond aux exigences suivantes du référentiel BTS SIO SISR :

Supervision : Mettre en place un système de monitoring complet permettant de détecter les pannes et anomalies sur l'ensemble du réseau.

Accès distant : Permettre aux collaborateurs externes de se connecter de manière sécurisée à l'infrastructure via un tunnel VPN.

Téléphonie : Déployer un système de communication interne permettant aux employés de s'appeler directement sur le réseau de l'entreprise.

Sécurité : Cloisonner le réseau en zones distinctes (LAN, DMZ, WAN) avec des règles de filtrage appropriées via pfSense.

Réalisé avec Nathan — Portfolio BTS SIO SISR 2025/2026

Déploiement d'un serveur VPN OpenVPN avec interface web Pritunl sur Ubuntu 24.04, hébergé en zone DMZ derrière pfSense. Le serveur permet aux collaborateurs distants de se connecter de manière sécurisée au réseau interne.

Architecture : Internet → Livebox (port 60165) → pfSense WAN (port 1194/UDP) → DMZ → Pritunl (192.168.100.101). Segmentation en 3 zones : WAN (192.168.1.0/24), LAN (192.168.2.0/24), DMZ (192.168.100.0/24), subnet VPN (10.10.0.0/24).

Audit pfSense : Identification et correction de failles critiques — mot de passe admin par défaut, règles DMZ trop permissives, SSH exposé sur Internet supprimé au profit d'un accès exclusif via VPN.

Problèmes résolus : Crash MongoDB (signal ILL) dû à l'absence du jeu d'instructions AVX en environnement Proxmox kvm64 — résolu par l'installation de MongoDB 4.4 + libssl1.1. Conflit réseau WireGuard résolu par désactivation du module.

Configuration : Chiffrement AES-256-GCM, hash SHA-256, DH 2048 bits, MTU optimisé (MSS/Fragment à 1400), buffers 384 Ko. Profils .ovpn générés par utilisateur avec certificats PKI individuels.

Optimisations : Ajustement MTU pour éviter la fragmentation, configuration Outbound NAT pour le subnet VPN, recommandation de passage en split tunnel pour les performances.

Ubuntu 24.04PritunlOpenVPN pfSense NATPKIAES-256-GCM

Réalisé avec Nathan — Portfolio BTS SIO SISR 2025/2026

Conception et déploiement d'un système de téléphonie IP complet sur Debian 12 avec FreePBX 17 et Asterisk 21, installés manuellement (sans ISO Sangoma). Le serveur est hébergé en LAN (192.168.2.8) sur Proxmox.

Installation manuelle : Compilation d'Asterisk 21, installation de MariaDB, configuration Apache avec l'utilisateur système asterisk, création du service systemd FreePBX — principal défi technique du projet.

Extensions SIP : 3 comptes créés en PJSIP (port 5060/UDP) — Alice (Grandstream physique), Bob (Linphone PC), Charlie (Softphone). Messagerie vocale activée avec accès via *97/*98.

Problèmes résolus : HTTP 500 (erreur VirtualHost Apache), permission denied sessions PHP (passage Apache en utilisateur asterisk), Unable to connect to Asterisk Manager (recréation du fichier AMI), absence de son RTP (ouverture des ports UDP 10000-20000 sur pfSense).

Accès VPN : Configuration NAT/RTP dans pjsip_custom.conf (local_net pour LAN et DMZ, external_media_address) pour permettre les appels depuis les clients connectés via le VPN Pritunl.

Résultats : Appels bidirectionnels fonctionnels sur le LAN avec audio, messagerie vocale opérationnelle. Configuration VPN → LAN en cours de finalisation pour le flux RTP.

Debian 12Asterisk 21FreePBX 17 PJSIPMariaDBRTP/SIP

Réalisé avec Nathan — Portfolio BTS SIO SISR 2025/2026

Mise en place d'un système de supervision complet avec Zabbix 7.0 et Grafana sur Ubuntu Server 24.04, hébergé en DMZ (192.168.100.102). Le projet a été réalisé en 3 niveaux progressifs couvrant la ligne internet, les équipements réseau et les postes de travail.

Niveau 1 — Ligne internet : Supervision du ping, de la latence, de la perte de paquets, du CPU, de la mémoire et du trafic réseau. Dashboard Grafana avec rafraîchissement automatique toutes les 10 secondes.

Niveau 2 — Switch D-Link : Intégration SNMP v2c du switch de la section SIO (192.168.1.243). Application du modèle D-Link DES/DGS, collecte du trafic par port, de la disponibilité et de la latence.

Niveau 3 — Postes et services : Déploiement de l'agent Zabbix 7.0 sur un PC Windows (171 métriques collectées). Supervision du service HTTP via vérification TCP sur le port 443.

Alertes email : Configuration SMTP Gmail avec mot de passe d'application. Déclencheur configuré sur le CPU (seuil > 60%) avec envoi automatique d'alertes. Test validé par charge artificielle PowerShell.

Migration DMZ : VM migrée en 192.168.100.102 pour respecter la segmentation réseau. Règles pfSense configurées sur les 3 interfaces (WAN, LAN, OPT1) pour autoriser les flux SNMP, Zabbix agent (10050/10051), HTTP, SMTP et ICMP.
